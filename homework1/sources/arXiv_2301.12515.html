LiDAR-CS Dataset: LiDAR Point Cloud Dataset with Cross-Sensors
for 3D Object Detection

Jin Fang1,2, Dingfu Zhou1, Jingjing Zhao1, Chulin Tang3, Cheng-Zhong Xu2 and Liangjun Zhang1 1 Robotics and Autonomous Driving Laboratory, Baidu Research.2 State Key Lab of IOTSC, CIS, University of Macau.3 University of California, Irvine.
Abstract

LiDAR devices are widely used in autonomous driving scenarios and researches on 3D point cloud achieve remarkable progress over the past years. However, deep learning-based methods heavily rely on the annotation data and often face the domain generalization problem. Unlike 2D images whose domains are usually related to the texture information, the feature extracted from the 3D point cloud is affected by the distribution of the points. Due to the lack of a 3D domain adaptation benchmark, the common practice is to train the model on one benchmark (e.g. Waymo) and evaluate it on another dataset (e.g. KITTI). However, in this setting, there are two types of domain gaps, the scenarios domain, and sensors domain, making the evaluation and analysis complicated and difficult. To handle this situation, we propose LiDAR Dataset with Cross-Sensors (LiDAR-CS Dataset 111Dataset link: https://github.com/LiDAR-Perception/LiDAR-CS), which contains large-scale annotated LiDAR point cloud under 6 groups of different sensors but with same corresponding scenarios, captured from hybrid realistic LiDAR simulator. As far as we know, LiDAR-CS Dataset is the first dataset focused on the sensor (e.g., the points distribution) domain gaps for 3D object detection in real traffic. Furthermore, we evaluate and analyze the performance with several baseline detectors on the LiDAR-CS benchmark and show its applications.

I Introduction

Recently, with the rapid development of deep learning (DL) techniques and the upgrade of hardware, DL-related applications have been largely promoted and Autonomous Driving (AD) is one of the most representative applications. Light Detection and Ranging (LiDAR) sensor, due to its stable performance on range measurement, becomes the dominant sensor for 3D perception tasks. To well promote the technique development, many AD-related 3D perception datasets have been released such as the KITTI [1], nuScene [2] and Waymo [3].

Although impressive performances have been achieved, there still remain some urgent problems required to be solved in the community. First of all, the across-dataset model evaluation is complicated. Usually, this kind of evaluation is necessary to verify the model’s generalization ability, however, this process is very complex and time-consuming due to several reasons: (1) different sensors. The LiDAR points are collected from different sensors, e.g., KITTI (Velodyne 64 beams LiDAR), nuScene (32 beams LiDAR); (2) different labeling rules. The labeling rule is different, in the KITTI dataset, only the objects in the front views have been annotated while the nuScene and Waymo are labeled in 360°view; (3) different categories. KITTI and Waymo share the same labeled categories while nuScene defines more specific categories. (4) different evaluation metrics. Last but not least, the evaluation metric for each dataset is also quite different which confuses the comparison over different datasets.

Secondly, various LiDAR sensors have been presented recently with different designs and functions with the rapid development of autonomous driving technology, such as Velodyne [4], Pandora [5] and Livox [6], For each manufacturer, there are lots of LiDAR sensor types. However, the existing LiDAR point cloud benchmarks are trying to enlarge the scenario diversity while the diversity of sensors has been ignored. But none of them provide more diverse data with different LiDAR sensors. As shown in Fig. 1, the detectors’ performances drop dramatically when the training and the testing are based on different sensors. Therefore, a dataset that can enlarge the sensor’s diversity will be very helpful for the community.

Thirdly, domain adaptation is another issue for point cloud perception research. Most previous works [7, 8] employed multi datasets for the research of domain adaptation. However, one fact is often being ignored that both the scenario domain and sensor domain exist across different datasets. The existence of two variables at the same time makes it difficult to evaluate and analyze different domain adaptation approaches. In general, it is easy to capture data from different scenarios by using the same sensor, but it seems difficult to collect data with exactly the same scenarios by using the different sensors in real application. That is because the background can keep the same while the foregrounds are dynamic all the time.

Motivated by the issues mentioned above, a novel LiDAR-CS (LiDAR-Cross-Sensors) dataset has been proposed here which can well address these issues. In general, compared with existing benchmarks, LiDAR-CS dataset has the following unique properties.

Cross Sensors. Currently, LiDAR-CS includes samples from 6 types of LiDAR sensors. For each LiDAR frame, we have generated 6 duplicates with the same scenarios and the same foreground’s position, the only difference is the point distribution which comes from the inherent attributes of the sensors, such as the beam number, field of view, angle resolution, etc. In addition, the data can be easy to be extended by adding another LiDAR sensor while only needing to provide some real scanned frames.

Consistent Annotation. We generate all the sensors’ data with the same background and foreground, therefore, we can keep consistent annotation for all the sensors. With the same annotation, the evaluation metric therefore can keep the same for all the data.

II Related Work

In this section, we generally review recently released LiDAR point cloud datasets in autonomous driving as well as LiDAR simulation and perception techniques.

LiDAR Dataset for AD Perception. With the rapid development of DL and AD, various large-scale datasets have been released recently. For the 3D object detection task, the most popular benchmarks are KITTI [1], nuScenes [2], and Waymo [3]. KITTI is the first large-scale public dataset for perception tasks in real AD scenarios. The point cloud is collected by a 64-beams Velodyne (VLD) LiDAR device. Three categories have been annotated in this benchmark while only the objects visible in the front camera view have been annotated. NuScenes is another dataset collected by a 32-beam LiDAR device. The point cloud frames are fully annotated with a 360°°\degree view. Objects are divided into 23 classes with attributes like visibility. Waymo Open Dataset [3] was conducted using 1 mid-range LiDAR and 4 short-range LiDARs. The categories of objects are following KITTI, but the frames are annotated in the full 360° view.

Cross Domains Dataset. Domain (or Sensor) generalizability is a very important property for deep learning-based approaches. A lot of benchmarks have been released for the 2D image domain [9] while few works have been proposed for the 3D point cloud. In [10], a cross-sensor dataset has been proposed with two types of LiDAR sensors (e.g., HDL-64 and VLP-32) and the transferability of features extracted from different sensors are also discussed, but unfortunately, the dataset is not released to the public. Waymo [3] proposes a cross-scenario domain adaptation task, which spans data from three different cities and split the data into two different domains. Experiments show that remarkable domain gaps exist between those cities.

LiDAR Simulation. LiDAR simulation [11] [12] is very popular in the field of robotics and AD. [13] collects the calibrated images and point cloud using the APIs provided by the video game engine, and applies these data for vehicle detection in their later work [14]. CARLA [15] and AutonoVi-Sim [16] also furnish the function to simulate LiDAR points data from the virtual world. Unlike previous simulators that entirely rely on CG models and game engines, in [17], an augmented simulator based on the “scan-and-simulate” pipeline has been proposed. Specifically, a vehicle with a LiDAR scanner is deployed to sweep the street of interest to obtain the background points cloud and then the foreground objects (e.g., CAD models) are placed on the background. LiDARsim [18] leverage the multi sweeps to build the dense background and reconstruct the foreground from cropped point cloud instead of CAD models.

LiDAR-based 3D Object Detection. LiDAR-based 3D object detection has been well studied for many years. Generally, all these approaches can be divided into three categories by the way of LiDAR point cloud representation as 2D projection-based or view-based [19, 20, 21, 22, 23, 24], voxel-based [25, 26, 27, 28, 29] and raw point cloud-based [30, 31, 32, 33], etc. To evaluate the performance of existing detectors on our proposed cross-sensor benchmark, PointPillars, SECOND, PointRCNN, PV-RCNN and CenterPoint are utilized, the details of the detectors are demonstrated in Sec. IV-A.

III LiDAR-CS Dataset

In this section, we introduce the methodology and details of the LiDAR-CS Dataset. A pattern-aware LiDAR Simulator is designed to produce high-quality and realistic dataset which can be generally adapted to most of the LiDAR sensors by only providing several real scanned frames. The sensors employed and the settings for our dataset are provided in Section. III-B, followed by the details of the dataset and evaluation metric.

III-A Cross-Sensor Data Generation

As we have mentioned before, cross-sensor data with exactly the same scenarios are difficult to be collected in the real application, it is straightforward to employ the data simulation techniques to generate data. To remove the domain gap between real data and simulation data as much as possible, we utilize an augmented LiDAR simulator proposed in [17], which uses the scanned high precision point cloud as background and only synthesis foreground objects with CAD (Computer-Aided Design) models, and leverage the graphics rendering techniques to generate a realistic LiDAR point cloud. Experimental results show that the generated point cloud has a similar performance as real data for the 3D object detection task.

Furthermore, based on [17], we propose a pattern-aware LiDAR Simulation framework that is more general to suitable for different LiDAR sensors and combinations. The overview of the framework is shown in Fig. 2, which consists of two modules, the Pattern Generation module and the Data Generation module.

III-A1 Pattern Generation Module

For sensor simulation, parameters azimuth resolution, vertical angles, and spin rate are prerequisites. However, not all the LiDAR point clouds can find the corresponding parameters of the sensor, and multi-sensors equipment is common for nowadays AD applications, therefore it is complicated to search the parameters for all the sensors. Moreover, the parameters in the manual usually are physically different from the real applications.

To well handle this problem, we present a Pattern Generation module, which can recover the LiDAR ray patterns from the scanned LiDAR points automatically and the LiDAR parameters are not prerequisites anymore. We call the normalized point cloud as Ray Pattern which represents the distribution of the rays of this sensor. Note that some laser rays will not return (some points are missing) in a certain frame due to different reasons, therefore the Ray Pattern from one frame may be incomplete. To handle this, we propose to merge the ray patterns across multi-frames and remove the duplicate rays to get a complete Ray Pattern.

III-A2 Data Generation Module

With the estimated Ray Pattern, simulation LiDAR point cloud can be generated by leveraging the CG (Computer Graphics) techniques. We project the scene onto 6 faces of a cube centered at the LiDAR center to form the cube maps, the environment point cloud is rendered with surface splatting techniques to obtain the smooth and holeless projection while the obstacle CAD models are rendered with regular methods.

However, the rendering module is the speed bottleneck of the system during the data generation processing. For multi-sensors simulation, we can project all the Ray Patterns to the depth map together and query the depth values simultaneously for each sensor. By this technique, the time-consuming point cloud simulation can keep nearly constant with the increasing number of sensors. Since the location and attributes of the synthesis CAD models are predefined, the ground truth of the foreground objects is annotated automatically during the point cloud simulation processing.

III-B LiDAR-CS Dataset

Currently, 6 different types of LiDAR sensors have been simulated here. The detailed information for each LiDAR sensor is listed below.

  1. 1.

    Velodyne Puck 16 (VLD-16) has 16 channels and maximum 100m range with 30∘superscript3030^{\circ} vertical field-of-view (FOV).

  2. 2.

    Velodyne 32E (VLD-32) has 32 channels, and vertical FOV from −30∘superscript30-30^{\circ} to +10∘superscript10+10^{\circ}.

  3. 3.

    Velodyne HDL-64E (VLD-64) has 64 channels, and 120m sensing range with 26.9∘superscript26.926.9^{\circ} vertical FOV.

  4. 4.

    Velodyne 128 (VLD-128) can capture the range of maximum 245m, and vertical FOV from −25∘superscript25-25^{\circ} to +15∘superscript15+15^{\circ}.

  5. 5.

    ONCE-40 is used in ONCE dataset [34] which is a 40-beams LiDAR sensor with vertical FOV from −25∘superscript25-25^{\circ} to +15∘superscript15+15^{\circ}. However, the parameters of the sensor have not been provided in the original paper [34] which illustrates the necessity of our pattern-aware framework.

  6. 6.

    Livox. Different from the mechanical scanning LiDAR devices introduced above, Livox is a solid-state LiDAR with incommensurable retina-like [35] scanning pattern. We follow Livox Simu dataset [36] and remove the dynamic pattern character to simplify the simulation processing. The sensor contains five Horizon LiDAR devices and one Tele-15 LiDAR device.

All the sensors are set up at the same height (e.g.,, 2m in this work), and with 360∘superscript360360^{\circ} horizontal FOV.

III-C Dataset Details

We simulate 14,000 point cloud frames for each pattern from 6 different sensors. In total, LiDAR-CS Dataset contains 84,000 point cloud frames with 6 groups. We provide a semantic category and a cuboid bounding box as (x,y,z,w​i​d​t​h,l​e​n​g​t​h,h​e​i​g​h​t,y​a​w​a​n​g​l​e)𝑥𝑦𝑧𝑤𝑖𝑑𝑡ℎ𝑙𝑒𝑛𝑔𝑡ℎℎ𝑒𝑖𝑔ℎ𝑡𝑦𝑎𝑤𝑎𝑛𝑔𝑙𝑒(x,y,z,width,length,height,yaw~{}angle) for the foreground object annotation. For each group, the data is divided into two splits, 7,000 frames for training and 7,000 frames for testing. For better comparison, all the groups share the same data splitting. For the foreground objects, five categories have been annotated in this dataset as Car, Truck, Pedestrian, Bicyclist and Motorcyclist. To be clear, the Truck category includes not only trucks but also other big vehicles such as vans and buses, etc. The number of objects in each category has been provided in Tab. I. Although the object number for “Car” is much larger than other categories, the object numbers for the rest categories are relatively balanced.

III-D Evaluation Metric

We follow the evaluation metric in ONCE benchmark [34]. Mean average precision (mAP) is calculated over the five categories while the orientations of the objects are also considered, objects will be regarded as false positive when the orientation fails to fall in −90​°90°-90\degree to 90​°90°90\degree of the ground truth orientation. Only objects within 70m are evaluated in our metric.

IV Empirical Studies

On top of LiDAR-CS benchmark, we evaluate several 3D object detection baselines on the generalization ability and explore the influence of points distribution domain gap.

IV-A 3D Object Detection Baselines

To well analyze the performances of different detectors across different sensor data, we employ five state-of-the-art 3D object detection baselines here which include points-based, voxel-based, and anchor-free-based frameworks respectively. Details of these baselines are introduced below:

  • •

    PointPillars [30] transforms the points into vertical pillars to form the 2D feature map and the feature in each pillar is extracted with a light-weighted MLP network, then the 2D backbone is employed for the following detection task;

  • •

    SECOND [26] is an improved version of VoxelNet by introducing the sparse 3D convolution to replace the traditional 3D convolution which can boost both the 3D detection performance and efficiency;

  • •

    PointRCNN [37] is a purely points-based framework that utilizes PointNet++ as the backbone to extract features for the segmentation branch and the bounding box proposal branches respectively. In addition, the second refine module is also designed to refine the bound box proposal leverage the cropped point cloud together with the point features extracted from the first stage;

  • •

    PV-RCNN [38] is a hybrid approach that learns discriminative features by combining both the 3D voxel and points representation;

  • •

    CenterPoint [39] which is an anchor-free-based method and achieves state-of-the-art performance on various benchmarks.

IV-B Cross-Sensors Evaluation

Five mainstream detectors are employed here to reveal perception domain gaps between different LiDAR sensors. In our experiments, all the detectors are trained independently on the training split (2,000 frames of them are used as validation set during the training) of each point cloud group and evaluated on the testing split of all the groups, respectively.

The detailed evaluation results are shown in Tab. II. Limited by the space, the results for VLD-128 sensor data and categories of “Bicyclist” and “Motorcyclist” have not been given in this table, readers could refer to our dataset homepage for the complete evaluation results. To be clear, the mAP is still computed over all the five categories.

Domain Gaps from Points Distribution. From this table, we can obviously find some interesting observations. Firstly, the model trained and tested on the same sensor gives the best performance in most situations. This is easy for understanding because they come from the same sensor therefore the point cloud distribution is exactly identical. Secondly, the detectors trained on a low-resolution dataset can achieve comparable performance on the high-resolution testing dataset while the opposite is not true. Taking the detectors trained on VLD-32, they achieve similar performances by testing on VLD-32 and VLD-64 while the performances drop dramatically on VLD-16.  Thirdly, the domain gaps positively correlate to the similarity of point ray distribution. For example, models trained on VLD-32 data outperform models trained on VLD-64 data when testing on VLD-16 dataset, vice versa. Finally, we find that some sensors exhibit similar performance, such as ONCE-40 and VLD-64, the mAP results are very close whether as training data or testing data. This is because even though they have different beam numbers, but they share similar vertical angles’ distribution.

Detectors Evaluation. We also compare the performances of different detectors. Overall, PV-RCNN achieves the best performance on large obstacles such as “Car” and “Truck”, while CenterPoint outperforms other baseline detectors with a remarkable gap in the “Pedestrian” category. When facing the domain gap situation, the performances of all the baselines decrease dramatically, especially for PointRCNN which relies on a Points-based backbone for feature extraction and the extracted features will change dramatically with the points distribution changed. Relatively speaking, voxel-based methods have the ability to deal with the change of point distribution to some extent. Experimental results illustrate that mainstream baseline detectors do not have the points-distribution-insensitive ability. We propose the hypothesis that the extracted features from the detectors are disabled to learn the real structure of the objects or underlying surfaces, but are confined to learning the distribution of points.

IV-C Domain Alignment Evaluation

To handle the domain gap between different sensor data, an intuitive way is to align the laser beams from the source sensor to the target sensor. To our best knowledge, few approaches and datasets have been proposed to solve the cross-sensors gap problem. Two baseline domain alignment methods (e.g., ST3D [8] and SN [7]) are employed for evaluation here. SN takes the BBoxs’ size from the target dataset to normalize the BBoxs in the source dataset during the training while ST3dD uses a Random Object Scaling strategy to expand the diversity of object sizes.

For comparison, we propose a simple strategy, NNDS (Nearest-neighbor scan down-sampling), which reduces the domain of point distribution by downsampling the source point cloud along the target scan distribution, as in [40]. But different from [40] simply dropping the scan lines uniformly, NNDS utilizes nearest-neighbor scan searching with target data as a reference, and drops the other lines.

Tasks of  VLD-64 →→\rightarrow VLD-32 and VLD-64 →→\rightarrow VLD-16 are given in Tab. III. PointPillars and CenterPoint are employed here as baseline detectors. From the table, we can see that the baselines perform worse than the NNDS. Both ST3D and SN  are proposed for solving the domain gap that comes from the different scenarios. But our dataset is proposed to simulate the domain gap coming from different LiDAR sensors and the statistical information of objects from different sensors are the same. Therefore, our dataset introduces a new challenge for domain alignment.

V Potential Applications and Discussion

Besides the 3D object detection task, the proposed LiDAR-CS dataset can also be applied to follow applications potentially.

Data Domain Adaptation The domain gaps between different sensors, simulation and real-world, are very important topics for real AD applications. The proposed pattern-aware point cloud simulation technique can be employed for generating any type of LiDAR sensor.

Distribution-insensitive features. Our LiDAR-CS dataset could be applied to evaluate the anti-interference ability of features extracted from different backbones, as we showed in Tab. II. Furthermore, perception systems with sensor-invariant ability can also be evaluated.

Large-scale Point Cloud Up-sampling or Re-sampling. Existing methods for point cloud up-sampling are studied on toy point cloud samples from CAD models with only thousands of points, few works focus on the LiDAR point cloud up-sampling in large-scale autonomous driving scenarios. Our dataset provides simulated point clouds with real scenarios and large-scale frames with different resolutions, making it very suitable for point cloud up-sampling or re-sampling research.

Semantic Segmentation. Since the ground truth bounding box of all the objects is provided, the semantic label for foreground objects are also can be obtained easily and our dataset can also be applied to the semantic segmentation task.

Sensor Selection. Our methodology could also be applied for LiDAR hardware selection and parameters adjusting. Instead of intuitively selecting the LiDAR device type and the installation parameters, or the combination of devices, then collecting and annotating the data, and training the perception models, it will be safer and more scientific to evaluate the devices, settings with perception systems ahead using simulation data with low-cost.

VI Conclusion

In this paper, we propose LiDAR-CS Dataset, which fills in the missing benchmark on points distribution domain adaptation and sensor-insensitive detectors for 3D object detection. For generating LiDAR-CS Dataset, we present a pattern-aware LiDAR Simulator to simplify the calculation of sensor ray tracing and accelerate the data generation. Empirical experiments on baseline detectors across different sensor devices not only bring interesting conclusions and heuristic insight but also benefit future research. We have released the dataset to the public and hope it could show the potential to promote related researches on the 3D point cloud. In the future, LiDAR-CS dataset will support more LiDAR sensors and more perception tasks.
